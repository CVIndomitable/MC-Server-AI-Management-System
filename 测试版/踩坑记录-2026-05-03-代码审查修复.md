### 1. 后端 WebSocket 参数名错误导致客户端AI聊天崩溃

**位置**：`mc-admin-server/app/websocket/routes.py:188`

**问题**：
```python
ai_response = await ai_agent.process_message(
    user_message,
    server_id="client",
    status_data=None,  # ❌ 错误：参数名应该是 current_status
    query_only=True,
    ...
)
```

**原因**：
- `ai_agent.process_message()` 方法的参数名是 `current_status`，而不是 `status_data`
- Python 不会在调用时检查关键字参数名，导致运行时 TypeError

**修复**：
```python
ai_response = await ai_agent.process_message(
    user_message,
    server_id="client",
    current_status=None,  # ✅ 正确
    query_only=True,
    ...
)
```

**教训**：
- 重构时要全局搜索所有调用点，不能只改定义
- Python 的动态类型在这种情况下不如静态类型安全
- 应该添加单元测试覆盖所有 WebSocket 端点

### 2. 客户端 WebSocket 缺少 server_id 导致会话混淆

**位置**：`mc-admin-mod-forge-1.21.1/src/main/java/com/mcadmin/mod/client/ClientWebSocket.java`

**问题**：
- 所有客户端玩家共享同一个 `server_id="client"`
- 不同服务器的玩家对话历史会互相污染
- 玩家 A 在服务器 1 的对话可能被玩家 B 在服务器 2 看到

**修复方案**：
1. 在 `ClientWebSocket` 中添加 `serverId` 字段
2. 在 `client_hello` 消息中传递 `server_id`
3. 后端使用 `f"{server_id}:client"` 作为有效 server_id 隔离会话

**代码变更**：
```java
// ClientWebSocket.java
private volatile String serverId;

public void setPlayerIdentity(String uuid, String name, String serverId) {
    this.playerUuid = uuid;
    this.playerName = name;
    this.serverId = serverId;  // 新增
}

private void sendClientHello() {
    JsonObject hello = new JsonObject();
    hello.addProperty("type", "client_hello");
    hello.addProperty("player_uuid", playerUuid);
    hello.addProperty("player_name", playerName);
    hello.addProperty("server_id", serverId);  // 新增
    ...
}
```

```python
# routes.py
client_server_id = hello.get("server_id", "").strip()
effective_server_id = f"{client_server_id}:client" if client_server_id else "client"
```

**教训**：
- 多租户系统必须在设计阶段就考虑隔离性
- 会话管理要用复合键（admin_id + server_id）而不是单一键
- 测试时要模拟多用户、多服务器场景

### 3. 后端缓存命中后仍调用大模型浪费配额

**位置**：`mc-admin-server/app/services/ai_agent.py:298-310`

**问题**：
```python
cached_text = None
if not query_only:
    cached = await command_cache.get(...)
    if cached:
        cached_text = cached.get("text", "")
        logger.info("缓存命中...")

# ❌ 即使缓存命中也会继续调用大模型
model = self._resolve_model(...)
response = await provider_pool.call_with_failover(...)
```

**原因**：
- 非流式方法中，缓存只用于记录日志，没有提前返回
- 流式方法 `process_message_stream` 中正确实现了缓存命中直接返回
- 复制代码时遗漏了关键逻辑

**修复**：
```python
if not query_only:
    cached = await command_cache.get(...)
    if cached:
        logger.info("缓存命中，跳过大模型调用")
        # 重建 assistant_msg 并加入历史
        content_blocks = [...]
        self.conversation_history[hkey].append({
            "role": "assistant",
            "content": content_blocks,
        })
        return {
            "text": cached.get("text", ""),
            "tool_calls": cached.get("tool_calls", []),
            "cache_hit": True,
        }
```

**影响**：
- 每次缓存命中仍调用大模型，浪费 API 配额
- 响应时间增加 1-3 秒
- 成本增加约 50%（假设缓存命中率 50%）

**教训**：
- 缓存逻辑要在最外层，命中后立即返回
- 流式和非流式方法要保持一致的缓存策略
- 添加监控指标：缓存命中率、API 调用次数、成本

### 4. 模组线程安全问题：嵌套锁定

**位置**：`mc-admin-mod-forge-1.21.1/src/main/java/com/mcadmin/mod/client/ClientAiChat.java:159-173`

**问题**：
```java
synchronized (messageHistory) {
    if (!messageHistory.isEmpty()) {
        ChatMessage last = messageHistory.get(messageHistory.size() - 1);
        if (last.sender == Sender.SYSTEM && last.text.equals("AI正在思考…")) {
            messageHistory.remove(messageHistory.size() - 1);
        }
    }
    if (hasError) {
        addMessage(Sender.SYSTEM, errorMsg);  // ❌ addMessage 内部也会锁 messageHistory
    } else {
        addMessage(Sender.AI, text);
    }
}
```

**原因**：
- Java 的 `synchronized` 是可重入锁，不会死锁
- 但这种写法容易引起混淆，降低代码可读性
- 如果 `addMessage` 调用其他需要锁的方法，可能导致性能问题

**修复**：
```java
private void handleResponse(JsonObject json) {
    ...
    postToRender(() -> {
        removeThinkingMessage();  // 单独方法，内部加锁
        if (hasError) {
            addMessage(Sender.SYSTEM, errorMsg);  // 外部调用，各自加锁
        } else {
            addMessage(Sender.AI, text);
        }
    });
}

private void removeThinkingMessage() {
    synchronized (messageHistory) {
        if (!messageHistory.isEmpty()) {
            ChatMessage last = messageHistory.get(messageHistory.size() - 1);
            if (last.sender == Sender.SYSTEM && last.text.equals("AI正在思考…")) {
                messageHistory.remove(messageHistory.size() - 1);
            }
        }
    }
}
```

**教训**：
- 锁的粒度要尽可能小，避免嵌套
- 每个方法要么完全不加锁（内部方法），要么完全加锁（外部方法）
- 使用 `java.util.concurrent` 包的并发集合可以简化代码

### 5. iOS 超时时间硬编码导致扩展性差

**位置**：`mc-admin-ios/MCAdmin/Services/APIClient.swift:236`

**问题**：
```swift
func analyzeSparkArchive(...) async throws -> AnalyzeArchiveResponse {
    var req = URLRequest(url: url)
    req.timeoutInterval = 180  // ❌ 硬编码，其他请求用默认30秒
    ...
}
```

**原因**：
- 只有 `analyzeSparkArchive` 需要长超时，其他请求用默认值
- 如果未来有其他长时间操作（如大文件上传），需要重复写类似代码
- 超时时间应该是请求的属性，而不是特殊处理

**修复**：
```swift
private func request<T: Decodable>(
    _ method: String,
    path: String,
    body: (any Encodable)? = nil,
    query: [String: String]? = nil,
    timeout: TimeInterval = 30  // 新增参数，默认30秒
) async throws -> T {
    ...
    request.timeoutInterval = timeout
    ...
}

func analyzeSparkArchive(...) async throws -> AnalyzeArchiveResponse {
    try await request("POST", path: "...", timeout: 180)  // 调用时指定
}
```

**教训**：
- 避免硬编码，使用参数化配置
- 通用方法要支持可选参数，而不是为特殊情况写特殊方法
- Swift 的默认参数值非常适合这种场景

### 6. 模组重连延迟未重置导致长期断线后恢复慢

**位置**：`mc-admin-mod-forge-1.21.1/src/main/java/com/mcadmin/mod/client/ClientWebSocket.java:96`

**问题**：
```java
private static final long RECONNECT_RESET_THRESHOLD = 300000;  // 定义了但从未使用

@Override
public void onOpen(WebSocket ws) {
    currentReconnectDelay = INITIAL_RECONNECT_DELAY;  // ❌ 直接重置，没有考虑稳定性
    lastSuccessfulConnectionTime = System.currentTimeMillis();
    ...
}
```

**原因**：
- 指数退避策略：5s → 10s → 20s → 40s → 60s（最大）
- 如果连接稳定运行5分钟后断开，下次重连应该用 5s 而不是 60s
- 但代码中定义了阈值却没有使用

**修复**：
```java
@Override
public void onOpen(WebSocket ws) {
    long now = System.currentTimeMillis();
    // 如果距离上次成功连接超过阈值，说明连接稳定，重置延迟
    if (lastSuccessfulConnectionTime > 0 &&
        (now - lastSuccessfulConnectionTime) > RECONNECT_RESET_THRESHOLD) {
        currentReconnectDelay = INITIAL_RECONNECT_DELAY;
    }
    lastSuccessfulConnectionTime = now;
    ...
}
```

**教训**：
- 定义的常量要么使用，要么删除
- 指数退避要配合稳定性检测，避免"一次失败，永久惩罚"
- 网络重连策略要考虑短暂抖动和长期断线两种场景

### 9. 模组 Config.java Windows 权限收紧可能失效

**位置**：`mc-admin-mod-forge-1.21.1/src/main/java/com/mcadmin/mod/Config.java:76-84`

**问题**：
```java
try {
    Set<PosixFilePermission> perms = PosixFilePermissions.fromString("rw-------");
    Files.setPosixFilePermissions(path, perms);
} catch (UnsupportedOperationException | IOException e) {
    // Windows NTFS 回退到 File API
    boolean ok = file.setReadable(false, false)
                  && file.setReadable(true, true)
                  && file.setWritable(false, false)
                  && file.setWritable(true, true);
    if (!ok) {
        LOGGER.warn("Could not restrict config file permissions: {}", file.getAbsolutePath());
    }
}
```

**原因**：
- Windows 的 `File.setReadable/setWritable` 只能设置"所有用户"或"仅属主"
- 无法像 POSIX 那样精确控制 owner/group/other
- 如果同一台 Windows 机器上有多个用户，配置文件仍可能被读取

**建议**：
- 在文档中说明 Windows 用户需要手动设置 NTFS ACL
- 或者使用 JNA 调用 Windows API 设置 ACL（增加复杂度）
- 生产环境建议使用 Linux 服务器

**教训**：
- 跨平台代码要明确各平台的限制
- 安全相关功能要在文档中说明平台差异
- 不要假设所有平台都支持相同的安全特性

### 10. 模组 AiChatScreen 滚动方向可能不符合直觉

**位置**：`mc-admin-mod-forge-1.21.1/src/main/java/com/mcadmin/mod/client/AiChatScreen.java:213`

**问题**：
```java
scrollOffset += (int) (scrollY * SCROLL_SPEED / 10.0);
scrollOffset = Math.max(0, Math.min(scrollOffset, maxScroll));
```

**原因**：
- `scrollY > 0` 表示鼠标向上滚动
- `scrollOffset` 增加表示向上翻页（显示更早的消息）
- 但实际行为取决于 `maxScroll` 的计算方式

**建议**：
- 实际测试滚动行为，确保符合用户预期
- 如果方向反了，改为 `scrollOffset -= (int) (scrollY * SCROLL_SPEED / 10.0)`
- 添加注释说明滚动方向的逻辑

**教训**：
- UI 交互逻辑要实际测试，不能只看代码
- 滚动、拖拽等操作的方向要符合用户习惯
- 参考 Minecraft 原版聊天界面的滚动行为

3. **缓存策略**：命中后立即返回，不要继续调用后端
4. **线程安全**：避免嵌套锁，使用并发集合简化代码
5. **可配置性**：超时、阈值、限流参数都应该可配置
6. **跨平台兼容**：明确各平台的限制，在文档中说明差异
7. **代码清理**：定期删除废弃代码、空方法、未使用的常量
8. **实际测试**：UI 交互逻辑要实际测试，不能只看代码
