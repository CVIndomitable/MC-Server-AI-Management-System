## 1. 环境准备

### 必须安装

| 工具 | 版本要求 | 用途 |
|------|----------|------|
| Python | 3.11+ | 运行后端服务 |
| Node.js | 18+ | 运行跨平台客户端（可选） |
| Java | 21 | 构建 Minecraft 模组 |
| Docker + Docker Compose | 最新版 | 部署后端（推荐方式） |
| Xcode | 15+ | 构建 iOS 原生客户端（可选） |

### 需要准备的账号/密钥

- **大模型 API Key** — 用于 AI 对话功能，支持 Anthropic 原生 API 或兼容网关
- **一台云服务器** — 用于部署后端（1 核 1G 即可），需有公网 IP
- **Minecraft 1.21.1 服务器** — 需安装 NeoForge，能访问外网（模组会主动出站连接云服务器）

## 2. 部署后端服务

后端是整个系统的中枢，负责 AI 对话、指令中转、命令审核和用户管理。

### 2.1 获取代码

```bash
# 将 mc-admin-server 目录上传到你的云服务器
scp -r mc-admin-server/ user@your-server:/opt/mc-admin-server
```

### 2.2 配置环境变量

```bash
cd /opt/mc-admin-server

# 复制示例配置
cp .env.example .env

# 编辑配置文件
vim .env
```

`.env` 文件需要填写以下关键配置：

```properties
# 服务地址
HOST=0.0.0.0
PORT=8000

# JWT 密钥（随机生成一个长字符串）
SECRET_KEY=你的随机密钥字符串
ACCESS_TOKEN_EXPIRE_MINUTES=10080

# 大模型 API 配置
ANTHROPIC_API_KEY=你的API密钥
# 可选：如果使用兼容网关，设置 base_url
# ANTHROPIC_BASE_URL=https://your-api-gateway.com

# 三级模型配置（根据你使用的 API 网关填写对应模型名）
MODEL_FLASH=MiMo-V2-Flash
MODEL_STANDARD=MiMo-V2-Omni
MODEL_PRO=mimo-v2-pro

# Redis（使用 Docker 部署则保持默认）
REDIS_HOST=redis
REDIS_PORT=6379

# 模组认证 Token（自定义一个，模组端需要配置相同的值）
MOD_AUTH_TOKEN=你的自定义token

# 跨域配置（可选，多个域名用逗号分隔）
# CORS_ORIGINS=http://localhost:8081,https://your-domain.com
```

> **安全提醒**：`SECRET_KEY` 和 `MOD_AUTH_TOKEN` 请使用足够复杂的随机字符串，不要使用示例值。

> **模型说明**：系统支持三级模型路由——flash（简单操作，如执行命令）、standard（一般问题）、pro（深度分析诊断）。AI 会根据用户消息内容自动选择，客户端也可手动指定。`ANTHROPIC_BASE_URL` 可配置为任何兼容 Anthropic API 的网关地址。

### 2.3 Docker 部署（推荐）

```bash
# 启动后端和 Redis
docker compose up -d

# 查看运行状态
docker compose ps

# 查看日志
docker compose logs -f api
```

启动成功后，后端会监听 `http://your-server-ip:8000`。

### 2.4 本地部署（开发/调试用）

```bash
# 安装依赖
pip install -r requirements.txt

# 本地需要单独启动 Redis
# macOS: brew install redis && redis-server
# Linux: sudo apt install redis-server

# 启动后端
python run.py
```

### 2.5 验证后端是否正常

浏览器打开 `http://your-server-ip:8000/docs`，如果能看到 Swagger API 文档页面，说明后端启动成功。

## 3. 安装 Minecraft 模组

模组安装在你的 MC 服务器上，负责执行指令和上报状态。

### 3.1 构建模组

```bash
cd mc-admin-mod

# 构建
./gradlew build
```

构建产物位于 `build/libs/mcadmin-mod-1.0.1.jar`。

### 3.2 安装到 MC 服务器

```bash
# 将 jar 文件复制到 MC 服务器的 mods 目录
cp build/libs/mcadmin-mod-1.0.1.jar /path/to/minecraft-server/mods/
```

### 3.3 配置模组

首次启动 MC 服务器后，模组会在 `config/` 目录下生成默认配置。你也可以提前手动创建：

```bash
# 在 MC 服务器目录下创建配置
mkdir -p config
vim config/mcadmin.properties
```

配置内容：

```properties
# 云服务器后端的 WebSocket 地址
ws.url=ws://your-server-ip:8000/ws/mod

# 认证 Token（必须和后端 .env 中的 MOD_AUTH_TOKEN 一致）
ws.token=你的自定义token

# 服务器标识（用于区分多个 MC 服务器）
server.id=srv_001

# 状态上报间隔（毫秒），默认 5 秒
status.report_interval=5000

# 危险操作是否需要二次确认
security.require_confirmation=true
```

> **重要**：`ws.token` 的值必须和后端 `.env` 中的 `MOD_AUTH_TOKEN` 完全一致，否则连接会被拒绝。

### 3.4 启动 MC 服务器

正常启动你的 MC 服务器即可。查看服务器日志，如果看到类似以下内容说明模组连接成功：

```
[MCAdmin] WebSocket connected to ws://your-server-ip:8000/ws/mod
[MCAdmin] Status reporting started (interval: 5000ms)
```

## 4. 启动客户端

系统提供两种客户端：**iOS 原生客户端**（推荐 iOS 用户使用）和 **Expo 跨平台客户端**（支持 iOS/Android/Web）。

### 方案 A：iOS 原生客户端（推荐）

使用 SwiftUI 开发的原生 iOS 应用，体验更流畅。

#### 4A.1 配置后端地址

打开 `mc-admin-ios/MCAdmin/Services/APIClient.swift`，修改 `baseURL` 为你的后端地址：

```swift
private let baseURL = "http://your-server-ip:8000"
```

#### 4A.2 构建到手机

1. 用 Xcode 打开 `mc-admin-ios/MCAdmin.xcodeproj`
2. 选择你的开发者团队（Signing & Capabilities）
3. 连接 iPhone，选择目标设备
4. 点击运行（Cmd+R）

### 方案 B：Expo 跨平台客户端

支持 iOS、Android 和网页端，适合快速测试或 Android 用户。

#### 4B.1 安装依赖

```bash
cd mc-admin-client
npm install
```

#### 4B.2 配置后端地址

```bash
# 复制示例配置
cp .env.example .env

# 编辑配置
vim .env
```

```properties
EXPO_PUBLIC_API_URL=http://your-server-ip:8000
EXPO_PUBLIC_WS_URL=ws://your-server-ip:8000/ws
```

将 `your-server-ip` 替换为你的云服务器公网 IP。

#### 4B.3 启动开发服务器

```bash
# 网页版（推荐先用这个测试）
npm run web

# 或启动 Expo 开发服务器（支持手机扫码预览）
npm start
```

网页版启动后，浏览器会自动打开 `http://localhost:8081`。

#### 4B.4 手机端

```bash
# iOS 模拟器
npm run ios

# Android 模拟器
npm run android

# 或使用 Expo Go App 扫描终端中的二维码
npm start
```

## 5. 首次使用

### 5.1 创建管理员账号

后端首次启动时会自动创建默认管理员账号：

- **用户名**：`admin`
- **密码**：`admin123`

> **安全提醒**：登录后请立即在「设置」页面修改密码。管理员还可以在设置页创建新用户。

### 5.2 选择服务器

登录后进入**服务器选择**页面：

- **我的服务器**：显示你已绑定的服务器列表，点击即可进入管理
- **未绑定**：显示已连接但尚未绑定管理员的服务器，管理员可点击「绑定」成为该服务器的主管理员

> **多人协作**：其他用户绑定已有主管理员的服务器时，会自动提交申请，主管理员在服务器选择页可审批（批准/拒绝）。

### 5.3 聊天界面

选择服务器后进入聊天界面，这是你与 AI 管理员交互的主要方式。你可以用自然语言描述你想做的事情，AI 回复支持 Markdown 格式渲染（加粗、列表、代码块等）。

**示例对话：**

```
你：服务器现在状态怎么样？
AI：当前服务器运行正常，TPS 19.8，在线玩家 3 人（Steve, Alex, Notch），
    内存使用 2048MB / 4096MB。

你：把 Steve 设为管理员
AI：已执行 /op Steve，Steve 现在是服务器管理员了。

你：服务器有点卡，踢掉所有人然后重启
AI：⚠️ 检测到高风险操作（涉及 @a 全体选择器 + 服务器重启），需要你确认：
    1. 踢出所有在线玩家（3人）
    2. 重启服务器
    确认执行吗？
```

### 5.4 状态面板

切换到「状态」标签页，可以实时查看：

- **TPS** — 服务器每秒 Tick 数，正常值为 20，低于 15 显示警告
- **CPU 占用** — 系统总 CPU 使用率、MC 进程 CPU 使用率、CPU 核心数
- **内存占用** — 图表展示内存使用趋势
- **在线玩家列表** — 当前所有在线玩家
- **连接状态** — 模组在线/离线、AI 后端在线/离线指示

数据由模组每 5 秒自动上报更新。MC 服务器离线时会提示「数据可能过期」。

### 5.5 快捷操作

切换到「操作」标签页，提供常用的一键操作按钮：

- **重启服务器** — 安全关闭并重启（需确认）
- **立即备份** — 触发世界存档备份
- **发送公告** — 向全服玩家广播消息（需输入内容）
- **白名单添加** — 将玩家加入白名单（需输入玩家名）
- **停止服务器** — 关闭服务器（需确认）
- **保存世界** — 立即执行 save-all

危险操作（如重启、停止服务器）会弹出确认对话框。

### 5.6 设置页

切换到「设置」标签页，可以管理：

- **账号信息** — 查看用户名、系统角色、服务器权限
- **修改密码** — 修改自己的登录密码
- **多账号管理** — 保存多个账号，快速切换（按最近使用排序）
- **服务器管理**（主管理员）：修改服务器名称、查看/解绑管理员
- **用户管理**（系统管理员）：新增用户、重置密码、删除用户
- **退出登录**

## 6. 功能详解

### 6.1 AI 支持的管理操作

通过自然语言，AI 可以帮你执行以下操作：

| 操作类型 | 示例指令 | 对应 MC 命令 |
|----------|----------|--------------|
| 执行命令 | "把时间设为白天" | `/time set day` |
| 管理权限 | "给 Steve op" | `/op Steve` |
| 移除权限 | "取消 Steve 的op" | `/deop Steve` |
| 踢出玩家 | "踢掉 Alex" | `/kick Alex` |
| 封禁玩家 | "把 Griefer 封了" | `/ban Griefer` |
| 解封玩家 | "解封 Griefer" | `/pardon Griefer` |
| 白名单 | "把 Steve 加入白名单" | `/whitelist add Steve` |
| 广播消息 | "告诉所有人10分钟后重启" | `/say 10分钟后重启` |
| 修改天气 | "把天气改成晴天" | `/weather clear` |
| 传送玩家 | "把 Alex 传送到 Steve 那里" | `/tp Alex Steve` |
| 查看状态 | "服务器现在怎么样" | 返回 TPS/CPU/玩家/内存信息 |

### 6.2 智能模型路由

系统根据消息复杂度自动选择模型：

| 模型等级 | 触发条件 | 适用场景 |
|----------|----------|----------|
| Flash | 简短操作指令 | "踢掉Steve"、"设为白天" |
| Standard | 含"为什么""怎么办"等关键词，或长消息（>80字） | "怎么回事"、批量操作 |
| Pro | 含"诊断""排查""性能调优"等关键词 | "分析TPS低的原因"、崩溃诊断 |

仅查询模式下固定使用 Flash 模型。客户端也可在聊天时手动指定模型等级。

### 6.3 仅查询模式

在聊天界面可以切换为仅查询模式。此模式下 AI 不会执行任何操作，只会：
- 分析你的需求
- 给出具体的 MC 指令建议
- 解释指令的作用和注意事项

适合在不确定操作后果时先咨询 AI。即使 MC 服务器离线，仅查询模式也可正常使用。

### 6.4 命令审核系统

系统采用三层混合审核机制，防止误操作：

**第一层：规则引擎（毫秒级）**

根据命令类型自动评估风险：

| 风险等级 | 命令示例 | 处理方式 |
|----------|----------|----------|
| 低风险 | list、say、time、weather、save-all | 自动放行 |
| 中风险 | op、deop、kick、give、tp、gamemode、whitelist | AI 复核（如启用） |
| 高风险 | ban、pardon、@a/@e 全体选择器、重启服务器 | 需人工确认 |

**自动升级规则：**
- 使用 `@a` 或 `@e` 全体选择器 → 强制升为高风险
- 30 秒内连续 3+ 条命令 → 升为中风险（防误操作）
- 30 秒内涉及 3+ 个不同玩家 → 升为高风险
- `give` 数量超过 1000 → 升为高风险

**第二层：AI 复核（中风险命令）**

调用大模型二次确认：检查意图是否匹配、参数是否合理、有无隐藏风险。

**第三层：人工确认（高风险命令）**

命令暂存到 Redis，等待用户在客户端确认（120 秒超时自动取消）。

### 6.5 命令缓存

系统会自动缓存最近的命令结果（Redis，LFU 淘汰策略）。相同的消息在缓存有效期内（默认 1 小时）会跳过大模型调用，直接执行之前的操作，响应更快。

### 6.6 命令白名单

出于安全考虑，模组只允许执行以下命令：

`list`, `say`, `op`, `deop`, `kick`, `ban`, `pardon`, `whitelist`, `save-all`, `time`, `weather`, `gamemode`, `tp`, `give`

不在白名单内的命令会被拒绝执行。如需扩展，修改 `mc-admin-mod` 中 `CommandExecutor.java` 的白名单配置。

### 6.7 多轮对话

AI 会记住对话上下文（每个服务器最多保留 50 条消息），支持连续追问：

```
你：服务器里有谁在线？
AI：当前在线 3 人：Steve, Alex, Notch。

你：把第一个人踢了
AI：已踢出 Steve。

你：再把第二个也踢了
AI：已踢出 Alex。
```

### 6.8 三级记忆系统

AI 拥有长期记忆，分三个层级：

| 记忆类型 | 字数限制 | 说明 |
|----------|----------|------|
| 全局记忆 | 1500 字 | 所有服务器共享的运维经验 |
| 管理员记忆 | 1500 字 | 针对特定管理员的偏好和习惯 |
| 服务器记忆 | 1000 字 | 针对特定服务器的配置和历史 |

记忆会自动注入 AI 的 System Prompt，使 AI 能够记住跨会话的上下文。

**智能特性：**
- **自动标签分类** — 性能优化、玩家管理、模组配置等 7 大类，按标签智能检索相关条目
- **版本控制** — 保留最近 10 个版本，支持回滚
- **后台整理** — 定期自动合并相关记忆条目，优化检索效率

可通过 API 管理记忆条目。

### 6.9 用户与权限体系

**系统角色**：
- `admin` — 系统管理员，可创建/删除用户、绑定未绑定的服务器
- `user` — 普通用户，可申请绑定服务器

**服务器角色**：
- `owner` — 主管理员（首个绑定者），可审批申请、解绑用户、重命名服务器。owner 无法自行解绑，防止服务器变成无主
- `admin` — 普通管理员（经 owner 审批加入），可正常使用管理功能

### 6.10 安全机制

- **Token 认证** — 模组与后端之间的 WebSocket 连接需要 Token 验证
- **JWT 认证** — 客户端与后端之间使用 JWT Token（默认 7 天过期）
- **密码加密** — bcrypt 哈希存储，支持修改密码和管理员重置
- **命令白名单** — 只有预定义的安全命令才能执行
- **三层命令审核** — 规则引擎 + AI 复核 + 人工确认，高风险操作层层把关
- **权限隔离** — 用户只能管理自己绑定的服务器，服务器角色实时查询

## 7. 常见问题

### 模组连接不上后端

**检查清单：**

1. 确认 `mcadmin.properties` 中的 `ws.url` 地址正确，包含端口号
2. 确认 `ws.token` 与后端 `.env` 中的 `MOD_AUTH_TOKEN` 一致
3. 确认云服务器防火墙开放了 8000 端口
4. 确认 MC 服务器能访问外网（`ping your-server-ip`）
5. 查看 MC 服务器日志中的错误信息

### 客户端无法登录

1. 确认后端地址配置正确（iOS 原生版检查 `APIClient.swift`，Expo 版检查 `.env`）
2. 确认后端服务正在运行（`docker compose ps`）
3. 确认使用了正确的账号密码（默认 `admin` / `admin123`）
4. 检查浏览器控制台是否有 CORS 错误（Web 版）

### AI 回复"无法连接到服务器"

1. 确认模组已成功连接后端（后端日志中能看到 WebSocket 连接记录）
2. 确认 `.env` 中的 `ANTHROPIC_API_KEY` 配置正确且有余额
3. 确认模型名称配置正确（`MODEL_FLASH`、`MODEL_STANDARD`、`MODEL_PRO`）
4. 检查后端日志：`docker compose logs -f api`
5. 如果只是想咨询而非执行操作，可切换到「仅查询模式」，即使服务器离线也能使用

### 服务器选择页看不到服务器

1. 确认模组已成功连接后端（连接后服务器会自动注册）
2. 未绑定的服务器只有系统管理员（admin 角色）能看到
3. 下拉刷新列表

### 状态面板不更新 / 显示"等待服务器数据"

1. 确认模组 WebSocket 连接正常（后端日志可查看连接状态）
2. MC 服务器离线时状态面板会显示「数据可能过期」，这是正常行为
3. 确认客户端网络连接正常
4. 尝试刷新页面或切换标签页重新加载

### 命令审核超时

1. 高风险操作需要在 120 秒内确认，超时会自动取消
2. 如果频繁触发审核，检查是否短时间内发送了过多命令（30 秒内 3+ 条会触发防误操作保护）

### iOS 原生客户端构建问题

- 确保使用 Xcode 15 或更新版本
- 在 Signing & Capabilities 中配置好开发者团队
- 真机调试需要在 iPhone 上信任开发者证书（设置 → 通用 → VPN与设备管理）
- 确认 `APIClient.swift` 中的后端地址正确

### Expo 跨平台客户端构建问题

- 确保使用 Expo SDK 54 及对应版本的依赖
- iOS 需要在 Xcode 中配置好开发者证书
- Android 需要配置好 Android SDK 环境
- 使用 `npx expo-doctor` 检查依赖兼容性
- 详见 [Expo 官方文档](https://docs.expo.dev)

### 模组构建失败

- 确认安装了 Java 21（`java -version`）
- 确认 Gradle Wrapper 存在（`ls mc-admin-mod/gradlew`）
- 如遇网络问题，配置 Gradle 代理或使用镜像仓库
- 首次构建需下载大量依赖，请耐心等待
