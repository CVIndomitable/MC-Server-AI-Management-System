# 账号权限层级说明

## 双层角色体系

本系统采用 **系统角色 + 服务器角色** 的双层权限模型。

```
系统层：  admin（系统管理员）  >  user（普通用户）
                                    │
服务器层：                    owner（服务器拥有者）  >  admin（服务器管理员）
```

---

## 一、系统角色

| 角色 | 说明 |
|------|------|
| `admin` | 系统管理员，首次启动自动创建默认账号 `admin/admin123` |
| `user` | 普通用户，只能由系统管理员创建 |

### 系统管理员独有权限

- 创建 / 删除用户
- 查看用户列表
- 重置任意用户密码
- 查看未绑定的服务器列表
- 首次绑定服务器（将无主服务器分配给某人）
- 访问全局记忆

### 所有用户共有权限

- 登录 / 修改自己的密码
- 申请绑定已有服务器
- 管理自己的管理员记忆

---

## 二、服务器角色

用户通过「绑定」与服务器建立关联，绑定后获得服务器角色。

| 角色 | 层级值 | 说明 |
|------|--------|------|
| `owner` | 2 | 服务器拥有者，首个绑定者自动获得 |
| `admin` | 1 | 服务器管理员，通过审批加入后获得 |

### owner 独有权限

- 审批 / 拒绝绑定申请
- 移除服务器管理员
- 重命名服务器
- owner 不可自行解绑（防止服务器变成无主状态）

### owner + admin 共有权限

- 查看服务器成员列表
- 发送 AI 对话 / 执行服务器命令
- 查看服务器实时状态
- 确认 / 拒绝高风险命令
- 读写服务器级记忆

---

## 三、服务器绑定流程

### 首次绑定（无主服务器）

```
系统管理员 ──调用绑定接口──► 检测服务器无绑定 ──► 自动成为 owner
```

- 只有系统管理员可以绑定无主服务器
- 第一个绑定的用户自动获得 `owner` 角色

### 后续绑定（已有 owner）

```
用户提交申请 ──► pending（等待审批）──► owner 审批 ──┬─ 通过 → 创建绑定（角色=admin）
                                                     └─ 拒绝 → 申请关闭，可重新提交
```

- 每个（用户, 服务器）只允许一个 pending 状态的申请
- 被拒绝后可以重新申请
- 审批通过后角色固定为 `admin`，不会自动升级为 `owner`

## 五、认证机制

- **JWT Token**：登录后签发，默认有效期 7 天
- Token 载荷包含：`username`、`role`（系统角色）、`exp`（过期时间）
- 每次请求校验签名 + 确认用户仍存在于数据库
- 服务器角色通过查询 `user_servers` 表实时判断，不写入 Token

## 六、数据库表结构

### users（用户表）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自增主键 |
| username | TEXT UNIQUE | 用户名 |
| hashed_password | TEXT | bcrypt 加密密码 |
| role | TEXT | 系统角色，默认 `user` |
| created_at | TEXT | 创建时间 |

### servers（服务器表）

| 字段 | 类型 | 说明 |
|------|------|------|
| server_id | TEXT PK | 服务器标识 |
| name | TEXT | 服务器名称 |
| created_at | TEXT | 创建时间 |
| last_seen_at | TEXT | 最后在线时间 |

### user_servers（绑定关系表）

| 字段 | 类型 | 说明 |
|------|------|------|
| username | TEXT PK | 用户名（外键） |
| server_id | TEXT PK | 服务器ID（外键） |
| role | TEXT | 服务器角色：`owner` / `admin` |
| bound_at | TEXT | 绑定时间 |

### bind_requests（绑定申请表）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自增主键 |
| username | TEXT | 申请人（外键） |
| server_id | TEXT | 目标服务器（外键） |
| status | TEXT | `pending` / `approved` / `rejected` |
| created_at | TEXT | 申请时间 |
| resolved_at | TEXT | 处理时间 |
| resolved_by | TEXT | 处理人 |

## 七、关键保护机制

1. **owner 不可自行解绑** — 防止服务器变成无主状态
2. **owner 不可删除自己的账号** — 同上
3. **绑定申请去重** — 同一（用户, 服务器）只允许一个 pending 申请
4. **审批后角色固定** — 通过审批的用户只能获得 `admin`，不会自动变成 `owner`
5. **首次启动自动创建管理员** — 确保系统可用，默认密码应立即修改
